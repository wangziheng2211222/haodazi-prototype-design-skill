# 好搭子原型设计

## 目标

执行好搭子原型设计工作流：把产品需求和可选视觉参考，转成一个完整、可评审、可点击的多页面高保真原型。核心竞争点是工作流质量，不是泛泛地生成几个页面。

使用本技能时，目标产物应成为产品、设计、研发共同评审的对象：页面地图、可点击页面导航、高保真页面、状态覆盖、评审问题和交付说明。

## 执行原则

- 把需求文档作为业务逻辑最高优先级来源。
- 把截图作为视觉语言和组件模式证据。
- 把 Figma 作为可选的还原/设计系统证据通道，不把它当作默认主输入。
- 对模糊需求默认使用主动假设补全：替用户补齐合理业务闭环，而不是先追问一堆问题。
- 默认生成完整可评审结果；当用户要求先评审模块、方案或分阶段确认时，走模块确认路径，不跳过确认点。
- 内部拆解默认不打断用户，但要保留为结构化产物。
- 高保真必须继承已确认或已推断的模块拆解、`pageId` 和交互蓝图。
- 生成一个连贯产品，而不是几个孤立页面。
- 保留需求追踪：每张需求卡都要能回溯原始需求文本和受影响页面。
- 按需加载产品级设计规范，不要回退到通用后台模板。
- 按需加载产品交互蓝图，用它判断模块归属、入口位置、导航层级、Tab 关系和弹窗/抽屉承接；设计规范只负责视觉样式，不替代交互蓝图。
- 保持“截图风格复刻、Figma 还原、评审修复、交付资产”是不同业务能力，不要混成一个模糊步骤。
- 质量检查和修复要定向到 `pageId`、页面区域、字段、状态或交互，不要借修复整页重做。
- 优先保证可评审性，少做无依据装饰。
- 只有当缺失信息会导致无法判断产品类型、平台形态或核心业务方向时，才最多问一个阻塞澄清问题。

## 运行时边界

需要生成可运行原型代码时，使用 Vue 3 + Element Plus + Tailwind CDN。生成稿里不要使用 React、JSX、shadcn、`className` 或 `useState`。

生成 Vue 页面时：

- 用 `// ---TEMPLATE---` 和 `// ---SCRIPT---` 分隔代码。
- 通过全局 `Vue` 对象访问 Vue API，例如 `Vue.ref()`、`Vue.computed()`。
- 使用 Element Plus 组件，例如 `el-button`、`el-table`、`el-dialog`、`el-drawer`、`el-form`、`el-tabs`、`el-empty`。
- 提示反馈用 `ElMessage`，确认弹窗用 `ElMessageBox.confirm`。
- 使用真实感中文或领域相关 mock 数据。避免 `Lorem ipsum`、`test`、`abc` 和 emoji。

## 工作流

### 1. 输入整理

收集并归一化可用输入：

- PRD、功能说明、会议纪要、粘贴需求或产品想法。
- 截图、已有页面、品牌/风格参考或产品截图。
- Figma 链接、指定画框、node ID 或导出的设计上下文。
- 用户约束：目标用户、平台、必做页面、输出位置、保真度和评审重点。
- 产品线线索，例如蝉圈圈、蝉妈妈、蝉镜或其他内部产品。

如果用户只有模糊想法，先推断一个紧凑但完整的产品范围，并输出主动假设卡。主动假设卡不是提问表，而是说明“我将按以下方向生成；如需调整直接说，否则继续”。

当任务提到已知产品线、要求遵循设计规范，或没有截图/Figma 但需要产品原生视觉风格时，读取 `references/product-design-systems.md`。
当任务提到已知产品线，或需要判断功能入口、页面归属、模块落位、导航层级、Tab 关系、弹窗/抽屉入口时，读取 `references/product-interaction-blueprints.md`。
当任务包含模糊需求、需求补全、截图、Figma、模块评审、方案确认、评审修复、视觉 QA、交付资产，或不确定应该走哪条好搭子业务能力时，读取 `references/capability-lanes.md`。

默认不要因为“模块选择”而停下来，也不要因为信息不完整就连续追问。只有输入冲突导致业务目标不清、Figma 文件很大但没有目标画框，或用户明确要求先看拆解、模块评审或方案选择时，才停下等待用户确认。

### 2. 主动假设补全

当用户输入很短、业务不完整或不知道怎么描述需求时，先替用户补齐可执行的默认版本。

主动假设卡包含：

- 产品类型：例如运营工作台、销售 CRM、商家后台、内容审核台、数据看板。
- 主要角色：默认选择最可能的 1-3 类用户。
- 核心流程：用一条业务闭环表达从开始到完成的路径。
- 页面范围：列出默认会生成的页面和不会生成的边界。
- 关键状态：补齐空态、异常、权限、审批、失败、禁用、成功反馈等评审状态。
- 结构与视觉依据：说明使用需求、截图、Figma、产品交互蓝图、产品设计规范或通用行业推断。

表达方式：

```text
我先按以下方向生成：
- 产品类型：销售团队客户管理工作台
- 主要角色：销售、销售主管
- 核心流程：录入客户 → 跟进客户 → 转化成交
- 页面范围：客户工作台、客户详情、跟进记录、数据看板、规则设置
- 关键状态：空数据、跟进中、已转化、无权限、删除确认

如果方向不对，直接告诉我；否则我会按这个继续。
```

不要把主动假设卡写成问卷。用户不回复时，按假设继续生成，并把假设写入 `requirements-map.json`、`page-map.json` 和评审说明。

只有三类情况需要先问一个阻塞问题：

- 业务类型完全无法判断，例如用户只说“做一个系统”。
- 平台形态会显著改变结果，例如 Web 后台、小程序、移动端 App 无法判断。
- 输入互相冲突，例如要求“移动端小程序”但又要求复杂左侧导航后台。

### 3. 需求理解

提取：

- 产品类型和主要用户。
- 角色、权限、业务对象、字段、状态和规则。
- 核心用户流程和分支流程。
- 页面、弹窗、抽屉、表格、表单、详情、设置、空/错/加载态。
- 歧义、风险和缺失需求。

除非用户要求查看，否则这一步作为内部推理，不要打断输出。

为评审创建需求卡：

- 每张卡有稳定的 `requirementId`。
- 每张卡保留原始需求摘录或原文位置。
- 记录解析后的意图、验收标准、影响的 `pageId`、相关状态和未决问题。
- 在评审原型中，点击需求卡必须打开需求详情面板/页面，展示原文定位、生成页面/状态覆盖情况。
- 如果来源是长文档，用章节标题、段落序号、页码或附近短摘录作为原文定位。
- 对主动假设补齐的内容，标记为 `assumed`，并在评审问题里提示用户可调整。

### 4. 页面地图和 `pageId` 拆解

写页面前先创建稳定 `pageId`。同一评审面内的列表、详情、新增/编辑弹窗、抽屉、筛选和局部状态，应绑定到同一个 `pageId`。

如果加载了产品交互蓝图，先用蓝图判断：

- 功能归属哪个已有模块，不轻易新增顶级导航。
- 主入口放在左侧导航、顶部 Tab、筛选区、列表操作区、卡片操作区、详情区、弹窗或抽屉。
- 页面之间是同级导航、Tab 切换、跨页面跳转，还是在当前页用弹窗/抽屉承接。
- 蓝图未采集的入口、弹窗或抽屉内容，标记为 `assumed` 或待确认，不要伪装成已有功能。

默认页面结构：

- 产品需要入口时，生成总览或仪表盘。
- 为核心业务对象生成主列表/工作台页面。
- 生成详情页或详情抽屉。
- 生成新增/编辑流程。
- 如果需求涉及评审、审批、状态流转，则生成对应流程。
- 只有需求明确时，才生成设置或规则管理页。

不要为了显得丰富而制造冗余页面。

### 5. 可选模块确认路径

当用户要求“先看拆解”“模块评审”“方案选择”“优化此组”或需要先确认页面边界时，使用模块确认路径：

1. 输出模块评审状态：模块分组、模块标题、描述、优先级、变更类型、所属 `pageId`、关键字段/状态/流程、A/B 方案、风险和覆盖提示。
2. 保持同 `pageId` 模块联动选择，避免只选列表而漏掉详情、弹窗、抽屉或表单。
3. 生成交互蓝图：导航结构、页面层级、关键路径、跨页面跳转、通用交互、状态规则和视觉约束。
4. 生成高保真时严格继承已确认模块、`pageId` 和交互蓝图。

如加载了产品交互蓝图，模块确认路径里的交互蓝图必须记录来源，并说明哪些模块归属、入口位置或页面关系来自产品蓝图。

蓝图失败可以降级继续，但要记录“全局交互约束减少”的风险，不要把降级伪装成完整成功。

### 6. 视觉方向

如果提供截图，提取视觉 DNA：

- 布局密度、导航结构、信息层级和组件家族。
- 色彩、字体、圆角、阴影、描边、间距和数据密度。
- 主要交互模式和可见状态。
- 应避免的风格。
- 多截图时，明确主参考图、辅助参考图和冲突点。

如果提供 Figma 上下文，优先使用 Figma 节点、布局和 token 证据，不用截图猜测替代。若 Figma、截图、需求之间冲突，业务逻辑服从 PRD，精确视觉约束服从 Figma，整体氛围可参考截图。

需要精确产物结构时，读取 `references/output-contract.md`。最终 QA 或用户要求看评审问题时，读取 `references/review-checklist.md`。

如果加载了产品设计规范，要在 design tokens 中记录所用规范并说明 fallback。例如蝉妈妈暂无专属规范时，可以临时使用蝉圈圈规范，但必须说明这是 fallback。
如果加载了产品交互蓝图，要在 `interaction-blueprint.json` 或交付说明中记录所用蓝图、是否 fallback，以及蓝图用于哪些结构决策。蝉圈圈需求应同时按需加载蝉圈圈设计规范和蝉圈圈交互蓝图：前者决定视觉，后者决定功能放哪里。

### 7. 高保真生成

一次性生成所有已确定或推断的 `pageId`：

- 保持导航、命名、数据字段、状态标签和主操作一致。
- 包含真实感 mock 数据和可见状态样例。
- 在有评审价值的地方覆盖空态、加载态、错误态、禁用态、权限态、校验态和危险操作确认。
- 如果创建 HTML 评审壳，左侧作为实时/页面预览区域，右侧作为页面导航、需求卡或方案导航区域。
- 右侧导航保持简洁：页面、需求卡、关键状态、问题和交付说明。
- 支持点击需求卡后，右侧切换到带原文回溯的需求详情，同时保持当前页面预览可见。

如果是在已有仓库里工作，遵循该仓库技术栈和脚本。如果是独立产物，除非用户要求框架应用，否则优先生成可本地打开的单文件 HTML。

### 8. 评审和修复

交付前检查：

- 每条需求至少映射到一个页面、状态或评审说明。
- 每张需求卡都能打开详情，并回溯原始需求和生成覆盖。
- 主流程能跨页面走通。
- 状态名、字段和操作保持一致。
- 弹窗、抽屉、表单、表格、详情属于正确的 `pageId`。
- 如果加载了产品交互蓝图，功能入口、模块归属、导航层级、Tab 关系和弹窗/抽屉承接符合蓝图；偏离处有需求依据或评审说明。
- 多页面视觉语言一致。
- 生成的 Vue 代码符合运行时边界。
- 主动假设内容在页面、需求卡或评审说明里可追踪，不伪装成用户明确要求。
- 如果走模块确认路径，模块选择、交互蓝图和高保真页面之间没有断链。
- 从用户任务角度给出验收摘要：已覆盖模块、未覆盖模块、评审风险和建议下一步。

发现问题时，修复受影响页面，或明确记录评审问题。不要用“已完成”掩盖缺口。默认最多自动修复一轮；修复失败时说明原因和下一步。

处理用户反馈时，先分类：需求缺口、视觉偏差、交互问题、字段/状态问题、缺失页面、交付资产需求。除非问题真的改变共享产品结构，否则只修复受影响页面、区域、交互或样式 token。

### 9. 交付

返回原型时说明：

- 用户在哪里打开。
- 生成了哪些页面。
- 使用了哪些设计参考：截图、Figma 画框、产品设计规范、产品交互蓝图或 fallback。
- 使用了哪些主动假设，哪些假设需要用户后续确认。
- 产出了哪些交付资产，哪些有意跳过。
- 还剩哪些评审问题或假设。
- 做过哪些验证。
- 如果走模块确认路径，说明模块确认、质量检查、修复记录和生成记录。

如果用户要的是产物，直接创建文件，不要只描述计划。

分享或交付时不要包含用户自定义 API Key、Figma Token、支付信息、登录态、内部权限字段或企业未授权知识库内容。

## 输出模式

选择最小但有用的输出形式：

- **聊天蓝图**：用于早期头脑风暴或产品方向对齐。
- **独立评审原型**：用于快速交付仓库外的可打开产物。
- **仓库集成原型**：用于用户要求修改现有项目。
- **技能/插件演进说明**：用于继续改进这套工作流本身。

除非用户明确要求，不要写产品文档。头脑风暴阶段只在聊天里输出。
